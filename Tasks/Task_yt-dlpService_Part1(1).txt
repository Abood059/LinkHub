================================================================================
                مهمة برمجية: إنشاء YtDlpService ونماذج الملفات
        (وفق معايير فصل المسؤوليات المعتمدة في مشروع LinkHub)
================================================================================

الهدف:
تطوير خدمة `YtDlpService` المسؤولة عن التعامل مع أداة `yt-dlp` لتحميل الوسائط،
مع نماذج البيانات الداعمة (`FileStatus`, `BaseFile`, `VideoFile`, `AudioFile`).
يجب أن تلتزم الخدمة بشكل صارم ببنية المشروع الحالية، ولا تتجاوز نطاق مسؤولياتها.

================================================================================
القسم الأول: نطاق العمل والملفات المطلوب إنشاؤها/تعديلها
================================================================================

يُطلب من المبرمج تنفيذ الملفات التالية فقط، دون المساس بباقي أجزاء النظام:

1. `src/main/models/FileStatus.js`                     (جديد)
2. `src/main/models/BaseFile.js`                       (جديد)
3. `src/main/models/VideoFile.js`                      (جديد)
4. `src/main/models/AudioFile.js`                      (جديد)
5. `src/main/services/ytdlp.service.js`                (جديد)
6. تحديث `src/main/models/index.js`                    (إضافة exports)
7. تحديث `src/main/services/index.js`                  (إضافة export)

================================================================================
القسم الثاني: التصميم المطلوب لنماذج البيانات (Models)
================================================================================

[1] FileStatus.js
    - فئة تمثل الحالة الديناميكية لعملية تحميل.
    - الخصائص (جميعها اختيارية عند الإنشاء، بقيم افتراضية مناسبة):
        * percentage (Number)   : 0-100   (الافتراضي 0)
        * speed      (String)   : مثال "5.2 MiB/s" (الافتراضي null)
        * eta        (String)   : مثال "00:45"     (الافتراضي null)
        * downloadedBytes (Number) : (الافتراضي 0)
        * totalBytes      (Number) : (الافتراضي null)
        * isPaused        (Boolean): (الافتراضي false)

    - لا يحتوي على دوال معقدة؛ مجرد هيكل بيانات.

[2] BaseFile.js
    - فئة أساسية (Abstract عملياً) تورث منها VideoFile و AudioFile.
    - الخصائص:
        * id          (String)  : معرف فريد (UUID) يتم توليده داخل constructor إذا لم يمرر.
        * name        (String)  : اسم الملف النهائي بدون مسار (مثال "My Video.mp4")
        * extension   (String)  : امتداد الملف ('mp4', 'webm', 'mp3', 'm4a'...)
        * sourceUrl   (String)  : الرابط الأصلي
        * storagePath (String)  : المسار الكامل الذي سيحفظ فيه الملف
        * fileStatus  (FileStatus) : نسخة من كائن FileStatus (تُنشأ تلقائياً داخل constructor)

    - دوال مطلوبة:
        * toJSON()    : تعيد تمثيلاً بسيطاً (Plain Object) مناسباً للإرسال عبر IPC أو الحفظ.

[3] VideoFile.js (extends BaseFile)
    - خصائص إضافية (خاصة بالفيديو، يتم تعبئتها من inspectLink):
        * formatId   (String)   : معرف الصيغة في yt-dlp
        * resolution (String)   : مثال '1920x1080'
        * fps        (Number)   : مثال 30
        * codec      (String)   : مثال 'avc1' أو 'vp9'
        * width      (Number)   : (اختياري)
        * height     (Number)   : (اختياري)
        * fileSizeApprox (Number) : الحجم التقريبي بالبايت (إن وفرته yt-dlp)

    - يجب استدعاء super() مع تحديد type = 'video' أو تركه للفئة الأم.

[4] AudioFile.js (extends BaseFile)
    - خصائص إضافية:
        * formatId   (String)
        * abr        (Number)   : متوسط معدل البت بالـ kbps
        * codec      (String)   : مثال 'mp4a' أو 'opus'
        * fileSizeApprox (Number)

    - يجب استدعاء super() مع type = 'audio'.

[5] تحديث models/index.js
    - أضف:
        FileStatus,
        BaseFile,
        VideoFile,
        AudioFile

================================================================================
القسم الثالث: تصميم YtDlpService (الخدمة فقط)
================================================================================

**مبدأ أساسي**: الخدمة مسؤولة فقط عن:
- تحديد مسار أداة yt-dlp (مثل ScrcpyService).
- استدعاء ProcessManager لتنفيذ الأوامر.
- تحويل المخرجات النصية إلى تحديثات على كائنات FileStatus (بالمرجع).
- لا تقوم الخدمة بأيٍّ من:
    * التفاعل مع IPC.
    * الحفظ في قاعدة البيانات.
    * إرسال أحداث للنوافذ (هذا دور طبقة أعلى).
